# Triage - Pilot Targets & Seed User Plan

Goal: get real feedback from people who already feel crypto-tax reconciliation pain.
During Stage 1, the objective is proof and learning, not revenue maximization.

## Ideal Seed User Profile

### Crypto CPA Or Tax Firm

Best first pilot:

- already serves crypto clients
- does manual transaction cleanup today
- has clients with DeFi, bridges, multiple wallets, or old exchange history
- is willing to review an AI-generated workpaper rather than asking the agent to file
- can compare Triage output against their current workflow

Why this buyer matters:

- clear willingness to pay
- repeated workflow
- professional feedback
- lower trust barrier because the CPA remains accountable

### DeFi-Heavy Individual

Useful pilot profile:

- multiple wallets and chains
- has used Koinly, CoinTracker, TokenTax, or similar tools
- has unresolved self-transfer, bridge, LP, staking, or dead-exchange issues
- wants a CPA-reviewable workpaper, not autonomous filing

Why this user matters:

- fast access
- vivid demo stories
- can validate whether the output is understandable

## First Outreach Targets

### Tier 1 - Warm, Fast Feedback

- crypto CPAs in the operator's network
- tax preparers active in crypto Twitter, Telegram, Discord, or LinkedIn
- DeFi users in GOAT/bootcamp communities with prior tax-software pain
- friends-of-friends who can safely provide a redacted sample history

### Tier 2 - Structured CPA Pilots

- small crypto-specialist CPA firms
- firms that publish crypto tax guides or advertise DeFi tax help
- accounting firms that use existing software but still offer manual cleanup

### Tier 3 - Later Integrations

- tax workflow software serving CPA firms
- crypto portfolio/accounting platforms that need better reconciliation
- broker/exchange partners preparing for 1099-DA support

Do not start with enterprise integrations. First prove that the workpaper is useful.

## Pilot Offer

Suggested pilot ask:

> Give Triage one messy, redacted reconciliation case. We will reconstruct the ledger,
> mark every assumption, flag anything unreconciled, and return a CPA-reviewable workpaper.
> You tell us where the result is useful, wrong, unclear, or missing evidence.

For firms, ask for:

- one anonymized historical case
- their current time spent on cleanup
- the issues their current software failed to resolve
- permission to use aggregate, non-client-identifying metrics

## Pilot Success Metrics

- number of cases processed
- percentage of lots marked `RECONCILED`
- percentage marked `RECONCILED (assumption)`
- percentage and dollar exposure marked `UNRECONCILED`
- manual CPA review time saved
- number of self-transfers correctly matched
- number of broker/1099-DA mismatches explained
- number of critical findings added to memory
- number of corrections required after CPA review

## Feedback Questions

Ask every pilot reviewer:

- Would this workpaper save you time?
- Which parts would you trust immediately?
- Which parts need more evidence?
- Which assumptions should be worded differently?
- What output format would fit your workflow?
- What would make this worth paying for?
- What would block you from using it with clients?

## Demo-Day Pilot Narrative

The Stage 1 demo does not need a large pilot base. It needs a believable path:

1. show the messy-history fixture
2. show why ordinary import tools fail
3. show Triage's cited reconstruction
4. show the critical-finding note that improves future runs
5. show a small list of CPA/DeFi seed users to approach next

## Do Not Do Yet

- Do not claim Triage replaces CPAs.
- Do not ask users for seed phrases or write-capable keys.
- Do not ingest unredacted personal tax records into a public demo.
- Do not promise final tax liability.
- Do not optimize for broad consumer filing before the forensic wedge is proven.
